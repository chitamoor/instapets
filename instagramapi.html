Thoughts on the Instagram API and platform

General comments on the Instagram API

  • URLs follow a RESTful convention
  • The API exposes the underlying entities and relationships in an intuitive manner
  • Payloads are self descriptive, although they seem a bit more coarse grained for certain API end-points. Please see a suggestion - 6 in the Enhancements/Suggestions section
  • The error messages are clear
  • The API documentation is helpful to the point of eliminating any initial friction in getting started. Clear examples of the end points are provided. Having a sample JSON Payload/response for every end-point is very helpful

Enhancements/Suggestions for the Instagram API and platform

  1. Sorting options
    2. Sorting options in the API are pre-built. Would be good to expose them as URL params for more fine grained control
  2. Documentation
    3. Although all the API end points are well documented, it would be helpful to have a general section on the most common use-cases and the API endpoints the use-cases map to. For example, if a developer wants to find a list of all the media for a given hash tag (which I assume is a common use-case), he or she currently has to go through all the endpoint documentation looking for the intended one.

  3. Instagram App (consumer version)
    4. Although Instagram has gained masssive adoption primarily due to it's simple/minimal feature set, but IMO, the following feature set would greatly enhance utility without cluttering the app -
    • Re-gram. Similar to 'Re-tweeting'
    • Overlay. Being able to add (or overlay) text and links on to the media
  4. Instagram Enterprise App
    5. The Enterprise version of the app would be targeted specifically for brands, marketers and enterprises wanting to engage their audiences through the Instagram platform. IMO, it would make sense to roll out the "enterprise" feature set into a separate version of the app, thus preserving the simplicity of the consumer version. Basic feature set for the enterprise version -
    • Business/profile page (like FB Business page)
    • Being able to add (or overlay) text and links on to the media
    • Ability to create contests - both on the device and web
    • Ability to view and track various metrics - campaign, audience, contests etc - similar to FaceBook Insights

  5. Instagram Enterprise API

    6. Additional API endpoints for enterprise users.

    • Contests API
    • InstaInsights - Similar to FB Insights API
    • Programatically managing the Media on the site
      • Being able to create photos on the fly - will be useful to prepopulate a set of images for a given enterprise site
      • Being able to programmatically categorize/sort/group media
  6. InstaGraphAPI (similar to the FaceBook Graph API)

    7. Map the underlying Instagram entity relationships on to a Graph and expose that Graph through an API similar to the FaceBook Graph API. Having a single conceptual/programming model for all (or most ) of the FB APIs would greatly simplify the development efforts

  7. Query DSL or some subset of SQL (similar to FQL)

    • The current payload is pretty coarse-grained. Having the ability to only get what the developers need would substantially simplify code and also make the apps less prone to changes
    • Currently the developers have to programatically create the "join" functionality for traversing complex relationships. Providing a "join" functionality through a Query DSL would be great. e.g get a list of users that posted a media with a specific hashtag that have the most likes.
